The hard questions enterprise customers will ask — prepare precise, technically honest answers
must-have~40 min
▾
Enterprise platform architects and VPs of Engineering ask specific, pointed questions. The goal isn't a polished pitch — it's the kind of accurate, grounded answer that builds credibility with a technical audience who will immediately sense vagueness.
1
"What data leaves Figma and where does it go?" — Be specific: what the MCP server sends, where it goes (Anthropic's infrastructure), what Figma controls vs what Anthropic processes, and where enterprise DPA / BAA agreements apply. Vague answers fail with enterprise security teams.
2
"We already use Claude directly. What does adding the Figma MCP server give us?" — The answer is about structured design context: Claude without MCP is working from descriptions or screenshots. Claude with Figma MCP has component structure, token values, variant states, and hierarchy — the design system, not an approximation of it. That precision is what makes output accurate at scale.
3
"Our product teams are building more in code. How does Figma stay relevant in that workflow?" — The answer here is about what Figma adds at org scale: design system governance, consistency enforcement across 40+ product teams, and a shared source of truth that gives AI tools accurate context to work from. It's not about handoff — it's about what makes AI-generated output trustworthy across a large org.
4
"Can we self-host or run this air-gapped?" — Know the real constraints. What Figma controls, what's possible via local MCP setup in VS Code, what requires Anthropic infrastructure, and where the actual limits are today.
5
"How is this different from the Figma REST API — we already have integrations built on that?" — MCP adds a natural language interface and agent orchestration layer. The REST API gives you data on demand. MCP gives Claude the ability to reason about design decisions in context, across tools, without custom integration code per workflow.
How to use these: Write the answer you'd give if a staff engineer interrupts you mid-sentence and says "wait — how does that actually work?" You need to be able to go one level deeper on any of these, not repeat the same sentence louder.

Customer Scenario — Week 3
You're three weeks into an enterprise expansion deal at a 6,000-person financial services company. They're a strong Figma org — 200 designers, an established design system, and 40+ product teams. The SC on the original deal flagged that the VP of Platform Engineering, Raj, has become a blocker. Raj's team owns the front-end platform that all 40 product teams build on. In a recent exec QBR, Raj said two things: first, that his engineers have started prototyping directly in React using Claude and Cursor, and several teams have stopped using Figma as a handoff layer entirely. Second, that he evaluated Make and found the output quality "not production-ready" — his lead engineer ran a side-by-side with Claude Code and Make lost on fidelity, token accuracy, and component adherence.

You have a 45-minute call with Raj and his lead platform engineer, Sonia, next Tuesday. The account exec wants you to "re-establish the value of Figma in the AI workflow." Raj is not hostile — he's pragmatic. He'll engage honestly if you do. What do you do with these 45 minutes, and what do you actually show them?
Debrief prompts for the regroup
Did you try to re-sell Figma as a design tool — or did you meet Raj where he is and talk about what Figma adds at platform scale that code-first prototyping doesn't solve?
How did you handle the Make quality objection — did you concede it, reframe it, or demonstrate something that changed the framing?
What would Sonia (the lead engineer) need to see to walk away convinced — and is that different from what Raj needs?
If the Figma MCP server was the centerpiece of your demo — did it actually address Raj's concerns, or did it just show a cool feature?
What's the honest version of this conversation — where would you tell Raj Figma genuinely adds value, and where would you acknowledge the limits?
How to Work Through This
Framework — don't defend Figma, understand Raj's actual problem
read first
▾
The instinct in this scenario is to re-sell Figma. That's the wrong move with a stakeholder like Raj. Work through these before you decide what to build or show.
1
What problem is Raj actually solving? His engineers are faster without Figma in the loop. That's a real observation, not a misconception. What does Figma genuinely add at a 6,000-person org with 40 product teams that code-first prototyping doesn't? Be honest about this before the call.
2
What did Make lose on in his engineer's test? Fidelity, token accuracy, component adherence — these are specific. Don't dismiss them. Understand what conditions produced those results and whether the Figma MCP server changes any of them.
3
What does Sonia need vs what Raj needs? Raj needs a platform-level answer. Sonia needs a technical one she can verify herself. These require different things from you in the same 45 minutes.
4
What's your honest position? There are things Figma + MCP does better than Claude alone. There are also things where the gap is real. Where would you tell Raj Figma genuinely wins — and where would you acknowledge the limits rather than paper over them?
5
Now build your 45 minutes. Discovery, demo, or workshop? What specifically would you show with the Figma MCP server that addresses what Raj actually raised — not just what looks impressive?
The trap to avoid: Showing a Figma MCP demo that is technically impressive but doesn't address the Make quality objection or the code-first workflow shift. Raj will notice. Sonia definitely will. The demo has to connect directly to what they said.

Enterprise-realistic test prompts + what to document
all tiers~60–90 min
▾
Run these same three prompts against Make Kits, Templates, and npm packages. Use a Figma file that has an actual design system — design tokens, components with variants, real hierarchy. Don't test against a toy file.
A
Component with design tokens: "Generate a primary button that uses our design system's color tokens, with default, hover, focus, and disabled states. The output should reference token names, not hardcoded values." — This is where Make often loses vs Claude Code. Document what you get and how much manual cleanup a developer would need.
B
Complex enterprise component: "Generate a data table with sortable headers, row selection, inline status badges, and a bulk action toolbar. Match the component library patterns exactly." — How does Make handle structural complexity? Where does it diverge from the Figma source?
C
Multi-variant component from a design system: "Generate a form input with label, helper text, error state, and character counter — matching our design system's spacing, typography, and color variables." — This tests whether Make can faithfully translate a component that has a well-defined spec in Figma. Compare output to what Claude Code alone would produce from the same prompt.
